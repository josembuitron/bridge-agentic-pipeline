Purpose
What it does
The pipeline takes unstructured input (meeting notes, emails, requirement docs) and produces structured deliverables through a sequence of specialized agents. A human reviews each agent's output before the next phase starts.

Input

Meeting transcripts, email threads, chat messages, requirement summaries, uploaded files (PDF, PPTX, images). Any format works.

Process

Five sequential phases. Each phase has a dedicated agent: translate requirements, research technologies, design architecture, build the solution, validate quality. The human approves at each gate before the next phase begins.

Output

Two tracks: internal pipeline artifacts (full technical detail) and client-facing deliverables (sanitized, professional). Formats include HTML reports, PowerPoint, Word, Excel, and architecture diagrams.

Foundation
How agents work
Each agent is a Claude instance with a specific role, a set of tools, and defined boundaries. The system runs within a strict two-level hierarchy.

What is an agent?

A Claude instance configured for a specific task. It has its own instructions, a defined tool set (file reading, web search, code execution), and a fixed scope. Think of it as a specialist who only does one job well.

What is an orchestrator?

The main agent that manages the overall workflow. It decides which specialist to call next, passes context between them, and handles the human review gates between phases.

The two-level constraint

Claude's architecture allows exactly two levels. The orchestrator (level 1) spawns sub-agents (level 2). Those sub-agents cannot spawn their own sub-agents. There is no level 3. The orchestrator plans the full breakdown up front, since it cannot delegate delegation further down.

Why this matters: Phase 4 (Build) uses dynamic specialists created on demand per project. Each specialist is a level-2 agent, so the orchestrator must coordinate all of them directly.

Security
Shift-left security at every phase
Security is not a Phase 5 afterthought. Powered by 35 security skills from Trail of Bits — the gold standard in security auditing — plus semgrep, gitguardian, and per-slice scanning. Attack surface mapping starts in Phase 3.

PHASE 3 Architecture

🔍
entry-point-analyzer

⚠
insecure-defaults

SECURITY IS NOT A GATE — IT IS THE PIPELINE
Traditional pipelines run security scans at the end. Bridge embeds security into every phase: threat modeling at design time, per-slice SAST during builds, and comprehensive auditing at validation. Vulnerabilities caught in Phase 3 cost 100x less to fix than those caught in production.
Testing
Testing strategy — the Dev-QA loop
Every vertical slice follows a mandatory Dev-QA cycle. Tests are written before code (TDD), hardened after passing, and verified end-to-end before advancing.
BUILDTDD: RED-GREEN-REFACTOR
TESTvitest run
HARDENerror paths + boundaries
E2EPlaywright smoke test
VERIFYall tests pass?
NEXT SLICEor retry (max 3)
Test hardening categories

Error paths

API returns 500, database is down, input is null, network timeout, permission denied

Boundary conditions

Empty arrays, zero values, max integer, very long strings, Unicode characters, negative numbers

Concurrency

Two users call simultaneously, race conditions, deadlocks, resource contention

Invalid input

Wrong types, missing required fields, SQL injection attempts, XSS payloads

State transitions

Happy path after error, recovery from partial failure, idempotency checks

Philosophy: Specialists write both TDD tests and hardened tests. The Validator evaluates whether tests are sufficient — it never writes tests itself. Separation of concerns: builders build, reviewers review.

Quality
Quality assurance at every phase
No phase output reaches the user without passing through at least one quality gate. Three layers of review ensure problems are caught early and fixed cheaply.

PHASES 1-3 Ojo Critico (Critical Reviewer)

✓ Skeptical senior reviewer with default REJECT posture
✓ Challenges missed requirements, unverified claims, generic template fills
✓ Checks for unrealistic cost estimates and hidden dependencies
✓ Writes CRITICAL/WARNING/NOTE findings with PROCEED or BLOCKED verdict
✓ Checkpoint enforcement: pipeline cannot advance without review file

PHASE 4 Per-Slice Quality Gates

✓ Per-slice semgrep SAST scan on changed files
✓ Deviation rules: auto-fix bugs, STOP on architecture changes
✓ Stall detection: max 3 retries per slice, then escalate to human
✓ De-sloppify pass: dead code removal, naming fixes, YAGNI check
✓ Walking skeleton failures escalate immediately (architecture is wrong)

PHASE 5 Comprehensive Validation

✓ Goal-backward verification: work backward from business goal
✓ 6-pass code review (pr-review-toolkit): bugs, tests, errors, types, comments, simplify
✓ Trail of Bits security audit: static-analysis + supply-chain + differential + variant
✓ Stub detection: empty bodies, TODOs, orphaned components, unbound handlers
✓ BRIDGE alignment: does solution address root causes and move impact metrics?
✓ Structured feedback routing: issues routed back to responsible agent

REQs addressed / total REQs in Technical Definition
Test Pass Rate 25%

Security Score 20%

Documentation 10%
Documented APIs / total APIs + README quality
Configuration
Configuration system
Every project starts with a config.json that controls pipeline behavior: execution mode, model routing, quality gates, and workflow features. Fully customizable per project.
// pipeline/config.json { "mode": "interactive", // or "yolo" (auto-advance) "granularity": "standard", // coarse | standard | fine "model_profile": "balanced", "workflow": { "discuss_phase": false, // pre-analysis conversation "critical_review": true, // Ojo Critico at gates "plan_checker": true, // pre-build validation "de_sloppify": true, // post-build cleanup "mutation_testing": false, // Stryker for critical logic "visual_regression": false, // pixelmatch UI checks "auto_advance": false }, "parallelization": { "enabled": true, "max_concurrent_specialists": 3 }, "gates": { "after_translate": true, "after_research": true, "after_architecture": true, "per_slice": true, "after_validation": true }, "model_profiles": { "quality": { "architect":"opus", "validator":"opus", "builders":"sonnet" }, "balanced": { "architect":"opus", "validator":"opus", "builders":"sonnet" }, "budget": { "architect":"sonnet", "validator":"sonnet", "builders":"sonnet" } } }

Execution Modes

interactive — Human approval gates at every phase (default). yolo — Auto-advance through all phases, only stop on errors or failures.

Model Profiles

quality — Opus for architecture + validation. balanced — Opus for critical decisions, Sonnet for building. budget — Sonnet everywhere, Haiku for cleanup.

Granularity Control

coarse — Fewer slices, faster delivery. standard — Balanced thoroughness. fine — More slices, more review cycles, maximum rigor.

Quality Gate Toggles

Each gate can be individually enabled or disabled. Turn off per_slice for batch review. Disable plan_checker for simple projects. All gates ON for enterprise.
