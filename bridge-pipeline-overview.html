The Framework

Why some AI projects start with the wrong problem

Most teams jump to technology. "We need a dashboard" becomes a React vs Angular debate before anyone asks what problem are we solving. BRIDGE forces the conversation to start from the real business problem, not a premature technology choice.

B

Business challenge

Separate what was said from what is actually needed. Clients describe symptoms; BRIDGE finds the underlying problem that, if solved, eliminates the symptoms.

R

Root causes

Structured causal analysis using the Fishbone method: People, Process, Technology, Data, Environment, Measurement. No guessing, only evidence-based diagnosis.

I

Impact and symptoms

Quantify the cost of inaction. Revenue lost, hours wasted, customer churn, compliance risk. Numbers make the case for investment.

D

Data and context

Map the existing landscape: what systems exist, what data is available, what constraints apply. Preliminary in Phase 1, then validated against live documentation in Phase 2.

G

Generate use cases

Create 3-5 technical use cases mapped directly to root causes. Each use case solves a specific diagnosed problem, not a vague wish list item.

E

Evaluate feasibility

Score each use case on viability, complexity, timeline, and risk. Prioritize what delivers the most value with acceptable risk.

BRIDGE is not linear. Any phase can loop back when new information surfaces. Research reveals a constraint? Go back to requirements. Building uncovers a feasibility issue? Revisit the architecture. Human approval gates control every transition.

Critical thinking

CT frameworks are reference knowledge embedded in agent prompts at key decision points. They tell the agent which lens to apply when reasoning. They are not separate scripts or modules.

Fishbone / Ishikawa Phase 1

Categorizes root causes into 6 dimensions: People, Process, Technology, Data, Environment, Measurement. Prevents jumping to solutions before understanding causes.

Force-Field analysis Phase 2, 3c

Scores driving forces vs restraining forces (1-5) for each technology or methodology choice. Makes trade-offs explicit and quantifiable.

SCAMPER Phase 3

Substitute, Combine, Adapt, Modify, Put to other use, Eliminate, Reverse. Prevents over-architecture by asking "can we simplify this?"

Six Thinking Hats Phase 3c

6 perspectives (data, intuition, risks, benefits, creative, process) evaluate 24 development methodologies. Prevents single-perspective bias.

Abductive reasoning Phase 4

When data is incomplete, formulate 2-3 hypotheses ordered by plausibility. Implement the most plausible with a verification step to confirm.

Goal-backward verification Phase 5

Dialectical: what conditions must be TRUE for the business goal to be met? Then check whether the code actually establishes those conditions.

Ojo Critico Phases 1-3

Skeptical senior reviewer that challenges weak reasoning, missing logic, and unsupported assumptions. Default posture: REJECT. Combines Paul-Elder intellectual standards with Watson-Glaser evaluation.

Karpathy Loop Post-project

Logs every key decision with phase, CT framework, confidence, and human override. After 3+ projects, patterns emerge in insights.json and feed future methodology selection. The pipeline learns.

Harness engineering

Mechanical enforcement via hookify files that run automatically on every tool call. Unlike .md rules that depend on the model "remembering" them, hooks execute as regex pattern matching, catching violations even when context degrades.

Taint cleanup[EXTERNAL-UNVERIFIED] tags in deliverables/

Layered enforcement model

Hookify provides always-on baseline (warn mode). Project pre-commit hooks add stack-specific checks (lint, type check, test, secrets). Pipeline protection hooks in settings.json can escalate to BLOCKING mode. Three independent layers, each stronger than the last.

Security and validation

Phase 5 runs four sequential validation agents, each with a different lens. The security gate is BLOCKING by default: any critical finding prevents delivery until resolved.

Validator

Goal-backward verification. Stub detection (empty functions, TODO in production). Requirements traceability matrix. Default posture: REJECT until evidence says otherwise.

Code reviewer

Clean code, single responsibility, test quality, YAGNI violations. Anti-praise guard: "looks good" without evidence is not a pass. 6-pass review via pr-review-toolkit.

Security auditor (BLOCKING)

SAST via semgrep, secrets detection via gitguardian, dependency audit (npm audit / pip-audit), OWASP Top 10, insecure defaults. Critical finding = delivery blocked.

Adversarial verifier

Tries to BREAK the code by executing it. Boundary values, idempotency, type confusion, concurrent access. Anti-rationalization: "the code looks correct" is not verified.

Quality score

Composite score: requirements coverage (35%) + test pass rate (25%) + security score (20%) + code quality (10%) + documentation (10%). Score >= 0.80 = APPROVE, 0.60-0.79 = CONDITIONAL, < 0.60 = REJECT and route feedback to responsible agent.

Resilience

AI models degrade as conversations get longer. BRIDGE has 10 rules for context management, state persistence across sessions, and emergency recovery when things go wrong.

Context budget protocol

  • Never accumulate: write output to files, keep only 3-5 line summaries
  • Phase handoff via disk: pass file paths to agents, not inline content
  • Prompt size guard: each agent prompt must be under 750 words
  • Phase-based refresh: re-read core rules at key transition points
  • Context anxiety detection: if the model starts rushing or skipping steps, spawn a fresh agent

State and recovery

  • Pipeline state file: state.json tracks phase, specialist progress, last checkpoint
  • Git rollback: tags after each phase, "go back to Phase 2" restores from git
  • Flexible execution: run phases in any order, skip ahead, parallel with reconciliation
  • Rejection loops: max 3 retries per slice, feedback includes WHY previous attempt failed
  • Cross-session resume: reads state.json on startup, presents status with options

Self-improvement

BRIDGE does not just execute projects. It learns from them. The Karpathy Loop correlates decisions with quality outcomes across multiple projects.

Decision logging

Every key decision is logged to ct-decisions.json: which phase, which agent, which CT framework was used, the confidence level, and whether the human overrode the decision. Human overrides are the most valuable signal.

Karpathy Loop (evaluate.ts)

After each project, a correlation script analyzes: which CT frameworks correlated with highest quality? Did methodology selection improve outcomes? Where did humans override most? Patterns confirmed across 3+ projects become insights.

Client knowledge graph

Per-client isolated knowledge: past technology decisions, constraints, anti-patterns, preferences. Strict isolation between clients. Knowledge data never appears in deliverables. Consolidated via "bridge dream {client}".

Agent archival

Successful specialist agents are archived for future reuse. When a new project needs a similar specialist, the pipeline checks the library for proven agents with matching technology profiles.

Documentation and deliverables

Two parallel output tracks: internal pipeline artifacts for your team, and sanitized client-facing documents. Stop at any phase and get professional deliverables.

Internal output (pipeline/)

Full pipeline detail: config, state, cost log, technical definition, BRIDGE analysis, research report, solution proposal, build manifest, validation reports, CT decisions, error logs, lessons learned.

Client output (deliverables/)

Sanitized documents with zero AI/agent/pipeline references. Includes: solution proposal, architecture diagrams, interactive HTML report, deployment guide, API reference. Optional: PPTX, DOCX, XLSX with brand customization.

6-tier documentation access

Agents research using a tiered chain: Context7 (code libraries) → DeepWiki (GitHub repos) → crawl4ai (any online docs, free) → Playwright (interactive sites) → Context Hub (68+ curated APIs) → WebSearch (fallback).

Early exit deliverables

Stop at any phase: Phase 1 = Requirements Document, Phase 2 = Technology Assessment, Phase 3 = Solution Proposal (most common exit), Phase 4 = Progress Report, Phase 5 = Full Deliverable Set.

Sanitization

Before client delivery, all internal terminology is replaced: "agent" becomes "team member", "MCP server" becomes "integration connector", "Claude" becomes "our team", "pipeline" becomes "project phases". The work is presented as a professional human team effort.
