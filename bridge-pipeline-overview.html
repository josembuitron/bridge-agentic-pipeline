Phase 0: Initialization

Tool discovery, workspace setup, configuration, and optional discuss phase

Tool & Resource Discovery

  • Detects 13 MCP servers, 11 CLI tools, 8+ plugins
  • Auto-installs missing CLIs (crawl4ai, semgrep, pandoc, pptxgenjs, exceljs)
  • Smart plugin gap check against recommended list
  • Builds fallback chain: crawl4ai → Playwright → Context Hub → Context7 → WebSearch
  • Detects brand assets and templates

Input & Workspace

  • 3 input modes: paste text, file paths, interactive description
  • Workspace safety validation (rejects Temp, .claude, node_modules)
  • Persistent workspace via ~/.bridge-workspace
  • Client folder reuse: detects returning clients and existing projects
  • Understanding validation: mandatory confirm before folder creation

Configuration

  • Modes: Interactive / YOLO
  • Models: Quality / Balanced / Budget
  • Granularity: Coarse / Standard / Fine
  • Presets: api-integration, data-pipeline, dashboard, enterprise-feature, mvp-rapid
  • Security gate: Blocking / Advisory
  • Harness hooks opt-in: Off / Warn / Enforce

Discuss Phase optional

Resolve ambiguities before Phase 1. Locks decisions as non-negotiable constraints in 00-constraints.md. All downstream agents treat these as immutable.

Brownfield Detection Phase 0b

If user references existing codebase: structure analysis, language detection, architecture pattern detection (monolith/microservice/serverless), integration points, database schemas. Uses Serena MCP for code intelligence.

Client Knowledge Graph

Loads per-client knowledge for returning clients: past tech decisions, constraints, anti-patterns, preferences. Strict per-client isolation — never cross-references between clients.

The Pipeline Phases

Phase 1: Translate

BRIDGE B-R-I-D analysis. Extracts business challenges, root causes (Fishbone categorization: People/Process/Tech/Data/Environment/Measurement), impact metrics, data context.

Sonnet Ojo Critico Fishbone CT

Output: Technical Definition + BRIDGE Analysis

Phase 2: Research

Live documentation research via 6-tier access. D-validation of Phase 1 assumptions. Security & Taint Assessment: classifies data sources by trust level, maps critical sinks, tool risk scoring.

Sonnet Ojo Critico Force-Field CT

Output: Research Report + Taint Assessment + D-Validated BRIDGE

Phase 3: Architect

Complete solution design: architecture (Mermaid/Excalidraw), file manifest, specialist team, vertical slices. Applies SCAMPER (Substitute/Combine/Eliminate) to prevent over-architecture. Sections A-I including Security Guardrails and Project Hooks.

Opus Ojo Critico Plan Checker SCAMPER CT

Output: Solution Proposal (A-I) + Diagrams + Agent Specs

Phase 3c: Methodology Selection NEW

CT-driven selection of development methodology from 24 frameworks (11 traditional + 13 AI-powered). Uses Six Thinking Hats (6 perspectives) + Force-Field analysis on top candidates. Adjusts Phase 4 config.

Opus Six-Hats CT Force-Field CT

Output: Methodology Selection + CT Analysis + Config Adjustments

Phase 4: Build

Dynamic specialist agents execute vertical slices with TDD. Walking skeleton first. Per-slice: security scan (semgrep), structural linting, garbage collection. Abductive reasoning for incomplete data decisions.

Sonnet Semgrep per-slice Structural Linter Abductive CT

Output: Working code + tests + build manifest

Phase 5: Validate & Deliver

Goal-backward verification, 6-pass code review, SAST security, secrets detection, dependency audit, quality scoring. Up to 22 Trail of Bits skills. Decision logging and Karpathy Loop evaluation for self-improvement.

Opus BLOCKING gate 6-pass PR review Decision Log

Output: Validation report + client deliverables + lessons + insights

What CT is NOT

CT frameworks are reference knowledge in prompts, not separate scripts. No 7-module TypeScript toolkit. No prompt templates pretending to be computation. The LLM already knows these frameworks — it just needs to know when to use them.

24 Development Methodology Catalog

Traditional (11): Agile, Waterfall, Scrum, Kanban, DevOps, Lean, XP, RAD, FDD, Spiral, Hybrid

AI-Powered (13): Agentic AI-Driven, Hybrid VSM, Context-First, Platform Engineering, AgentSecOps, CTEM, Data-Centric AI, Human-in-the-Loop, Predictive Sprint, Shift Intelligence Left, Automated QC, AI Governance at IDE, Upskilling

Each has bridge_compatibility score (0-1), best_for project types, and config_adjustments that change Phase 4 behavior.

Self-Improvement (Karpathy Loop)

After each project:

  • Log all key decisions to ct-decisions.json
  • Correlate decisions with quality outcomes
  • Track human overrides (most valuable signal)
  • 3+ projects = confirmed pattern in insights.json
  • Future projects use insights to select better methodologies

Controlled by: program.md (editable — human decides what to evaluate)

Phase 4: Build Process Detail

Dynamic Agent Creation

Dependency resolution before spawn:

  • CLI tools → auto-install via setup script
  • npm/pip packages → install before spawning
  • MCP servers → add to tools or degrade gracefully
  • Trail of Bits skills → invoke and embed methodology
  • Scripts → create mock servers, data generators

Quality check before writing agent:

  • Clear, specific task (not vague)
  • ALL tools assigned (from tool-matrix.md)
  • Doc access chain included
  • Prompt < 2,000 tokens (~750 words)
  • Completion signal defined
  • Workflow pattern selected (sequential / safety gate / task-driven / routing)

Post-Build Checks (Per Slice)

Structural Linter (orchestrator, no LLM):

  • File manifest compliance (expected vs actual)
  • Import direction enforcement (layer violations)
  • Naming convention compliance
  • File size guard (>300 WARN, >500 ERROR)
  • Test file presence check

Error Enrichment: Failures include architecture context, fix steps, and don'ts from the solution proposal.

Dev-QA Loop Per Slice

BUILD (TDD: red → green → refactor)

↓
TEST (vitest / pytest / framework-specific)

↓
HARDEN (error paths, boundaries, concurrency)

↓
E2E (Playwright smoke test if frontend)

↓
SECURITY SCAN (semgrep per-slice)

↓
STRUCTURAL LINT (5 checks)

↓
BRIDGE_SLICE_COMPLETE

De-Sloppify + Garbage Collector

De-Sloppify: Dead code, unused imports, naming fixes, debug removal, YAGNI cleanup (Haiku model).

Garbage Collector (5 checks):

  • GC-1: Dead code / orphaned files detection
  • GC-2: Pattern consistency audit
  • GC-3: Architecture drift detection
  • GC-4: Documentation freshness
  • GC-5: Duplicate code detection

Specialist Archival

Successful specialists archived to .claude/agents/library/ with track record. Future projects glob the library for technology matches and reuse proven agents.

Phase 5: Validation & Security Detail

Validation Agents (4 sequential)

5.1a Validator — Goal-backward verification. Stub detection. Requirements traceability. BRIDGE alignment. Doc-Architecture sync checks (DOC_DRIFT). Verdict: APPROVE/REJECT

5.1b Code Reviewer — Clean code, SRP, test quality, YAGNI, error handling at boundaries, documentation. ESLint scan. Verdict: PASS/FAIL

5.1c Security Auditor (BLOCKING) — SAST (semgrep), secrets (gitguardian), dependency audit (npm audit/pip-audit), OWASP Top 10, insecure defaults. Verdict: SECURE/BLOCKED

5.1d PR Review (6-pass) — Code quality, test analysis, silent failure hunting, type design, comment accuracy, code simplification

Quality Score

requirements × 0.35 + tests × 0.25 + security × 0.20 + code_quality × 0.10 + docs × 0.10

≥ 0.80 APPROVE | 0.60-0.79 CONDITIONAL | < 0.60 REJECT

Decision Logging & Self-Improvement

ct-decisions.json: Every key decision with phase, agent, CT framework used, confidence, and human override status.

Karpathy Loop: evaluate.ts correlates decisions with outcomes. Patterns from 3+ projects feed into insights.json for future methodology selection.

Security Skills in Phase 5

Always:

  • static-analysis (CodeQL + Semgrep + SARIF)
  • supply-chain-risk-auditor (CVEs, typosquatting)
  • differential-review (code vs architecture drift)
  • spec-to-code-compliance (code implements spec?)
  • audit-context-building (ultra-granular analysis)
  • fp-check (false positive verification gate)

Conditional:

  • variant-analysis + semgrep-rule-creator (if vuln found)
  • agentic-actions-auditor (if GitHub Actions)
  • zeroize-audit (if crypto/secrets in memory)
  • constant-time-analysis (if timing-sensitive crypto)
  • building-secure-contracts (if blockchain)
  • firebase-apk-scanner (if Android+Firebase)
  • second-opinion (if external LLM CLI available)

Security Gate

BLOCKING (default): ANY critical finding = BLOCKED. Options: auto-fix, manual fix, accept risk (intentionally friction-heavy — per finding), abort.

Advisory: Findings logged, do not block delivery.

Rejection Routing

Structured feedback to responsible agent via feedback-routing.json. Max 3 fix attempts. Tracks CT decision that led to rejection. Improvements logged to improvements.tsv.

Harness Engineering

Two independent hook systems protecting code quality and pipeline integrity

Project Pre-Commit Hooks

Architect generates hooks for the project's tech stack in Section I. Walking Skeleton installs them in Slice 1.

3 modes: Off (default) | Warn (detect, never block) | Enforce (block on violations)

Taint Tracking & Tool Risk

External content classified by trust level:

  • TRUSTED: Internal, controlled sources
  • SEMI-TRUSTED: Partner APIs with SLA
  • UNTRUSTED: User input, public web, file uploads

Critical sinks mapped: SQL, file writes, command exec, HTML render, API calls. HIGH-risk integrations get architectural guardrails in Phase 3.

Pipeline Protection Hooks (Claude Code)

Optional safety net for the pipeline process itself:

HookDetects
Destructive Command Guardrm -rf, git push --force, DROP TABLE, kubectl delete
Secrets in Output GuardAWS keys, API tokens, passwords in Write/Edit
Scope Escape GuardFile writes outside project path

Exception: .claude/agents/ is excluded (BRIDGE creates specs there).

Structural Linter

Runs after every BRIDGE_SLICE_COMPLETE (orchestrator, no LLM):

  • File manifest compliance vs architecture
  • Import direction enforcement (layer violations)
  • Naming convention compliance
  • File size guard (300 WARN / 500 ERROR)
  • Test file presence

Error Enrichment Protocol: Failures include architecture context, intended patterns, fix steps, and anti-patterns from the solution proposal.

Resilience & Context Management

Context Budget Protocol (9 Rules)

  • R1: Never accumulate — write to disk, keep 3-5 line summary
  • R2: Phase handoff via disk, not context
  • R3: Specialist prompts < 2,000 tokens
  • R4: Orchestrator keeps only checkpoint summaries
  • R5: Large project detection (>5 specialists)
  • R6: Context-by-reference pattern for all agents
  • R7: Phase-based refresh (before Phase 4, every 2 specialists, before Phase 5)
  • R8: Prompt size guard (extract to temp file if >750 words)
  • R9: Emergency recovery (detect degradation, re-read core.md + state.json)

Cost Tracking

  • Per-agent estimation after every spawn
  • Optional budget cap with 80% warning, 100% pause
  • Opus $15/$75, Sonnet $3/$15, Haiku $0.80/$4 per 1M tokens
  • Logged to pipeline/cost-log.json

Pipeline State & Resumability

  • state.json: current phase, completed phases, specialist progress, last checkpoint
  • Resume reads state.json first, falls back to file scanning
  • Git tags after each phase for rollback (bridge/{client}/{project}/phase-{N})
  • Flexible execution: out-of-order phases, parallel execution with reconciliation, cross-session resume
  • Rollback: "Go back to Phase N" restores pipeline/ from git tag

Behavioral Guardrails

  • Tool denial: Fallback chain, never stop (mark unverified)
  • Analysis paralysis: 5+ reads without writing = must explain or BLOCKED
  • Deviation rules: Auto-fix bugs; escalate architecture; skip scope creep
  • Stall detection: Missing signals, error keywords, timeouts
  • Rejection loops: Max 3 retries per slice, max 2 validation cycles

Cross-Run Learning

  • Failures requiring 2+ attempts → generate lessons (max 5/run)
  • Client knowledge graph persists across projects (strict isolation)
  • Successful specialists archived for future reuse
  • Karpathy Loop: decisions → outcomes → insights across projects

Design Principles

  • On-demand loading: Read files only when entering that phase — never preload
  • Context-by-reference: Pass file paths to agents, not inline content
  • Human in control: Approval gates at every phase, stop anytime
  • Dual output: Internal pipeline/ (full detail) + client deliverables/ (sanitized)
  • Security shift-left: Semgrep per-slice in Phase 4, 22 ToB skills in Phase 5
  • Self-improvement: Karpathy Loop learns from every project
