# Harness Hooks Module

Two independent hook systems that reinforce BRIDGE's guardrails with deterministic enforcement:

1. **Project Hooks** — Pre-commit hooks generated by the Architect and installed in the built project
2. **Pipeline Hooks** — Claude Code hooks that protect the BRIDGE pipeline itself

Both systems default to **warn mode** (detect + report, never block) to prevent disrupting legitimate work.

---

## SAFETY: Three Operating Modes

| Mode | Behavior | When to use |
|------|----------|-------------|
| **`off`** | No hooks installed | When hooks cause issues or are not needed |
| **`warn`** (DEFAULT) | Hooks run but only print warnings, never exit non-zero | First time using hooks, or when uncertain |
| **`enforce`** | Hooks block on violations | After validating hooks work correctly in a project |

**Config:** `config.harness_hooks` in `pipeline/config.json`:
```json
{
  "harness_hooks": {
    "project_hooks": "warn",
    "pipeline_hooks": "warn"
  }
}
```

**Configuration precedence:** `config.json` is the authoritative source. The `BRIDGE_HOOK_MODE` env var exists ONLY as a temporary emergency override (higher priority than config when set). If used, the user should update config.json afterward to prevent drift.

**Emergency override:** If a hook blocks and the user needs to bypass:
- Project hooks: `git commit --no-verify` (standard git escape)
- Pipeline hooks: Change mode to `"off"` in config.json, or user denies the hook tool call
- Temporary override: `export BRIDGE_HOOK_MODE=warn` (overrides config until shell closes)

## Part 1: Project Pre-Commit Hook Generator

### When It Runs

The **Architect** (Phase 3) generates hook configuration as part of the solution proposal. The **Walking Skeleton specialist** (Slice 1, Phase 4) installs the hooks as part of initial project setup.

### What the Architect Specifies (Section I of Solution Proposal)

Add to `03-solution-proposal.md`:

```markdown
## I. Project Quality Hooks (Harness Engineering)

### Hook Mode: {warn | enforce} (from config.harness_hooks.project_hooks)

### Pre-Commit Hooks
Based on the technology stack:

#### For {stack_name}:
| Hook | Command | Purpose | Blocking in enforce mode? |
|------|---------|---------|--------------------------|
| {hook_name} | {command} | {purpose} | {yes/no} |
```

### Hook Templates by Stack

#### Node.js / TypeScript
| Hook | Command | Purpose | Blocking? |
|------|---------|---------|-----------|
| Lint staged | `npx eslint --fix {staged_files}` | Style enforcement | Yes |
| Type check | `npx tsc --noEmit` | Type safety | Yes |
| Test affected | `npx vitest related {staged_files} --run` | Regression prevention | Yes (enforce only) |
| Secret scan | `grep -rn "AKIA\|sk-\|password\s*=" {staged_files} \| grep -v "tests/\|\.example\|\.md\|docs/"` | Credential protection | Yes |
| File size | `wc -l {staged_files} \| awk '$1 > 500 {print "TOO LARGE: "$2; exit 1}'` | Maintainability | No (warn only) |

#### Python
| Hook | Command | Purpose | Blocking? |
|------|---------|---------|-----------|
| Lint | `ruff check --fix {staged_files}` | Style enforcement | Yes |
| Format | `ruff format --check {staged_files}` | Format consistency | Yes |
| Type check | `ty check {staged_files}` (or `mypy`) | Type safety | Yes (enforce only) |
| Test affected | `pytest {related_tests} --no-header -q` | Regression prevention | Yes (enforce only) |
| Secret scan | `grep -rn "AKIA\|sk-\|password\s*=" {staged_files} \| grep -v "tests/\|\.example\|\.md\|docs/"` | Credential protection | Yes |

#### Go
| Hook | Command | Purpose | Blocking? |
|------|---------|---------|-----------|
| Vet | `go vet ./...` | Bug detection | Yes |
| Lint | `golangci-lint run` | Style + bugs | Yes |
| Test | `go test ./...` | Regression prevention | Yes (enforce only) |
| Secret scan | `grep -rn "AKIA\|sk-\|password\s*=" {staged_files} \| grep -v "tests/\|\.example\|\.md\|docs/"` | Credential protection | Yes |

### Installation Script

The Architect specifies the hook installation in the Slice 1 (Walking Skeleton) task:

```bash
#!/bin/bash
# {project-path}/scripts/setup-hooks.sh
# Generated by BRIDGE Architect — Harness Engineering hooks
set -euo pipefail

# Read mode from config.json (authoritative source), env var as emergency override
if [ -f "pipeline/config.json" ]; then
  CONFIG_MODE=$(grep -o '"project_hooks"[[:space:]]*:[[:space:]]*"[^"]*"' pipeline/config.json | grep -o '"[^"]*"$' | tr -d '"' 2>/dev/null || echo "warn")
else
  CONFIG_MODE="warn"
fi
HOOK_MODE="${BRIDGE_HOOK_MODE:-$CONFIG_MODE}"

HOOKS_DIR=".git/hooks"

# Safety: rollback on failure
trap 'rm -f "$HOOKS_DIR/pre-commit"; echo "❌ Hook install failed — rolled back"' ERR

mkdir -p "$HOOKS_DIR"

cat > "$HOOKS_DIR/pre-commit" << 'HOOKEOF'
#!/bin/bash
# BRIDGE Harness Engineering — Pre-commit hook
# Mode: controlled by BRIDGE_HOOK_MODE env var (warn | enforce)
# Override: git commit --no-verify

HOOK_MODE="${BRIDGE_HOOK_MODE:-warn}"
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM)

if [ -z "$STAGED_FILES" ]; then
  exit 0
fi

ERRORS=0
WARNINGS=0

# --- Hook checks inserted here by Architect ---
# Each check follows this pattern:
#   Run check → capture exit code
#   If fail AND enforce mode → increment ERRORS
#   If fail AND warn mode → increment WARNINGS, print warning

# Example: ESLint check
if echo "$STAGED_FILES" | grep -qE '\.(ts|js|tsx|jsx)$'; then
  JS_FILES=$(echo "$STAGED_FILES" | grep -E '\.(ts|js|tsx|jsx)$')
  if ! npx eslint $JS_FILES --quiet 2>/dev/null; then
    if [ "$HOOK_MODE" = "enforce" ]; then
      echo "❌ HOOK BLOCKED: ESLint violations found"
      ERRORS=$((ERRORS + 1))
    else
      echo "⚠️ HOOK WARNING: ESLint violations found (non-blocking in warn mode)"
      WARNINGS=$((WARNINGS + 1))
    fi
  fi
fi

# --- End of checks ---

if [ "$HOOK_MODE" = "enforce" ] && [ $ERRORS -gt 0 ]; then
  echo ""
  echo "🛑 Pre-commit blocked: $ERRORS error(s). Fix before committing."
  echo "   Override: git commit --no-verify"
  echo "   Switch to warn mode: export BRIDGE_HOOK_MODE=warn"
  exit 1
fi

if [ $WARNINGS -gt 0 ]; then
  echo ""
  echo "⚠️ Pre-commit: $WARNINGS warning(s). Proceeding (warn mode)."
fi

exit 0
HOOKEOF

chmod +x "$HOOKS_DIR/pre-commit"
echo "✅ BRIDGE pre-commit hook installed (mode: $HOOK_MODE)"
echo "   Change mode: export BRIDGE_HOOK_MODE={warn|enforce}"
echo "   Bypass: git commit --no-verify"
```

### How Specialists Interact with Hooks

When a specialist agent commits code and the hook fires:

**In warn mode (default):**
- Hook prints warnings but allows the commit
- Warnings appear in the agent's output
- Orchestrator includes hook warnings in the structural linter summary at the approval gate

**In enforce mode:**
- Hook blocks the commit
- Agent sees the error message with "Override: git commit --no-verify"
- Per deviation rules: agent may auto-fix (lint errors) or escalate (architecture violations)
- Agent should try to fix the issue first, then commit again
- If agent cannot fix: it reports the block in its output and the orchestrator handles it

**Agent prompt addition** (embedded by orchestrator when hooks are installed):
```
## Pre-Commit Hooks Active
This project has pre-commit hooks in {mode} mode.
If a commit is blocked:
1. Read the error message — it includes the specific violation
2. Fix the violation in the relevant files
3. Stage and commit again
4. If you cannot fix it after 2 attempts: output HOOK_BYPASS_REQUIRED: {reason} and STOP.
   Do NOT use --no-verify without orchestrator approval.
Do NOT disable hooks. The orchestrator will handle bypasses at the approval gate.
```

## Part 2: Pipeline Protection Hooks (Claude Code Hooks)

These protect the BRIDGE pipeline itself during execution. They are Claude Code hooks that run within the harness.

### Design Principles
- **Warn-only by default** — never block without explicit user opt-in
- **Narrow scope** — only protect against clearly dangerous patterns
- **Self-documenting** — every block message explains what happened and how to bypass
- **User-controlled** — can be disabled via config.json or by denying the hook tool call

### Recommended Pipeline Hooks

#### Hook 1: Destructive Command Guard
**Event:** PreToolUse (Bash)
**Purpose:** Warn before destructive commands run during pipeline execution

```
Patterns detected:
- rm -rf (recursive force delete)
- git push --force (history rewrite)
- git reset --hard (discard changes)
- DROP TABLE / DROP DATABASE (data destruction)
- kubectl delete (infrastructure destruction)

In warn mode: prints "⚠️ Destructive command detected: {command}" and allows
In enforce mode: blocks with "❌ Destructive command blocked. Override: change harness_hooks.pipeline_hooks to 'warn' in config.json"
```

#### Hook 2: Secrets in Output Guard
**Event:** PostToolUse (Write, Edit)
**Purpose:** Warn if written files might contain hardcoded secrets

```
Patterns detected:
- AKIA[A-Z0-9]{16} (AWS access key)
- sk-[a-zA-Z0-9]{20,} (OpenAI/Stripe secret key)
- password\s*=\s*["'][^"']+["'] (hardcoded password)
- -----BEGIN (PRIVATE|RSA) KEY----- (private key)

In warn mode: prints "⚠️ Possible secret detected in {file}" and allows
In enforce mode: blocks write with "❌ Possible secret blocked. Review {file}:{line} before proceeding."
```

#### Hook 3: Scope Escape Guard
**Event:** PreToolUse (Write, Edit, Bash)
**Purpose:** Warn if an agent tries to write outside the project directory

```
Patterns detected:
- Write/Edit to path NOT under {project-path}/ AND NOT under .claude/agents/
- Write/Edit to paths containing ~/.claude/settings*, ~/.claude/hooks*, node_modules/
- Bash commands that cd to directories outside {project-path}/

NOTE: .claude/agents/ is EXCLUDED from scope guard — BRIDGE legitimately creates
specialist agent files there during Phase 4. Only .claude/settings* and .claude/hooks*
are protected (pipeline configuration files).

In warn mode: prints "⚠️ Operation targets path outside project: {path}" and allows
In enforce mode: blocks with "❌ Path escape blocked. Only {project-path}/ is writable during pipeline."
```

#### Hook 4: Composite Action Guard
**Event:** PreToolUse (Bash)
**Purpose:** Detect chained commands where ANY part is destructive, even if other parts are safe

```
A command like: echo "done" && rm -rf /tmp/important
...would bypass Hook 1 if it only checked the first command.

Detection: Split command by && || ; & | and check EACH part against
the destructive patterns from Hook 1. If ANY part matches, trigger.

In warn mode: prints "⚠️ Destructive command in chain: {dangerous_part}" and allows
In enforce mode: blocks with "❌ Chained command contains destructive operation: {dangerous_part}"
```

#### Hook 5: Written-File-Execution Guard
**Event:** PreToolUse (Bash)
**Purpose:** Detect when a recently written file is about to be executed, and verify its content

```
Scenario: A specialist writes scripts/deploy.sh containing "rm -rf /",
then runs "bash scripts/deploy.sh". Hook 1 doesn't catch this because
the Bash command itself looks safe.

Detection (two-step):
1. PostToolUse on Write/Edit: record the file path in a session variable
   (BRIDGE_RECENTLY_WRITTEN_FILES, append-only, comma-separated)
2. PreToolUse on Bash: if the command executes a recently written file
   (bash {file}, node {file}, python {file}, ./{file}), read the file
   and check its content against destructive patterns from Hook 1.

In warn mode: prints "⚠️ Executing recently written file {file} — content contains: {pattern}" and allows
In enforce mode: blocks with "❌ Recently written file contains destructive pattern. Review {file} before executing."

Note: Only applies to files written IN THIS SESSION. Pre-existing project
files are not checked (they were already reviewed by the user).
```

### Installation

Pipeline hooks are NOT auto-installed. The orchestrator presents them as an option during Phase 0 initialization:

```markdown
## Harness Engineering Hooks (OPTIONAL)

Pipeline protection hooks are available. They detect destructive commands,
hardcoded secrets, and scope escapes during the build phase.

Current mode: {config.harness_hooks.pipeline_hooks}

Options:
a) Enable in WARN mode (recommended) — detect and report, never block
b) Enable in ENFORCE mode — detect and block dangerous patterns
c) Keep OFF — no pipeline hooks
d) Skip (decide later)
```

If the user enables hooks, the orchestrator writes them to the project's `.claude/settings.local.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "bash {project-path}/scripts/bridge-pipeline-guard.sh pre-bash \"$TOOL_INPUT\""
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "bash {project-path}/scripts/bridge-pipeline-guard.sh post-write \"$TOOL_INPUT\""
      }
    ]
  }
}
```

The guard script is generated by the orchestrator and placed in `{project-path}/scripts/bridge-pipeline-guard.sh`. It reads `BRIDGE_HOOK_MODE` from the environment.

### Troubleshooting

If hooks cause problems during a pipeline run:

| Symptom | Fix |
|---------|-----|
| Agent cannot commit | `export BRIDGE_HOOK_MODE=warn` or `git commit --no-verify` |
| Agent's Write calls are blocked | Change `pipeline_hooks` to `"warn"` in config.json |
| False positive on secret detection | The pattern matched a test fixture or example — agent should note bypass in output |
| Hook script error (not a violation) | The hook script itself has a bug — set mode to `"off"` and report in pipeline error log |
| Hooks slow down pipeline | Skip `test affected` hook (most expensive), keep lint and secret scan |

### Removal

At any point, the user can:
1. Set `config.harness_hooks.project_hooks` to `"off"` → orchestrator skips hook generation
2. Set `config.harness_hooks.pipeline_hooks` to `"off"` → orchestrator removes Claude Code hooks
3. Delete `.git/hooks/pre-commit` in the project → removes project hooks
4. Delete hooks section from `.claude/settings.local.json` → removes pipeline hooks

## Integration Points

### Phase 0 (Initialization)
- Read existing `config.harness_hooks` from config.json
- If first run: present hook options to user (default: warn)
- If continuing: use existing config

### Phase 3 (Architect)
- Generate **Section I: Project Quality Hooks** in solution proposal
- Select hooks appropriate for the technology stack
- Specify hook mode from config

### Phase 4 (Build — Slice 1)
- Walking Skeleton specialist installs hooks via `scripts/setup-hooks.sh`
- Subsequent specialists work with hooks active
- Hook warnings collected by structural linter at each approval gate

### Phase 5 (Validate)
- Validator checks: are hooks installed and functioning?
- Security auditor: do hooks cover secret detection?
- If hooks caught issues during build: log in `pipeline/04-build-manifest.md`
